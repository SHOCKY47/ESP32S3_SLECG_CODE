# 02 — 带宽与组包分析

本文档基于工程现有配置，推导 500 Hz 单导 ECG 的蓝牙传输参数选型。

## 1. ECG 源数据量

| 参数 | 值 | 来源 |
|------|-----|------|
| 芯片 | ADS1291（单通道） | `ADS129X_CHIP` |
| 采样率 | **500 Hz** | `ADS129X_SAMPLE_RATE_HZ = 500` |
| RDATAC 原始帧 | 6 B/样本（STATUS 3 + CH1 3） | `ADS129X_FRAME_SIZE` |
| 应用层输出 | **int16** `ch1_value`（2 B/样本） | `ads129x_sample_t` |

### 1.1 为何选用 int16 而非 raw 24-bit

- 驱动已将 24-bit 补码符号扩展并右移 4 位压缩为 `int16_t`
- 与 [`ads129x.h`](../components/ads129x/ads129x.h) 输出一致，上位机无需重复换算
- 500 × 2 = **1000 B/s**，比 raw 24-bit（1500 B/s）节省 33% 带宽

### 1.2 纯 ECG 样本吞吐

```
500 samples/s × 2 bytes/sample = 1000 bytes/s
```

## 2. BLE 链路能力

基于 [`ble_slecg.c`](../components/ble_slecg/ble_slecg.c) 现有配置：

| 参数 | 值 | 说明 |
|------|-----|------|
| BLE 版本 | 4.2 | `CONFIG_BT_BLE_42_FEATURES_SUPPORTED` |
| ATT MTU | **517** | `esp_ble_gatt_set_local_mtu(517)` |
| 单次 Notify 有效载荷 | **≈514 B** | MTU − 3（ATT 头） |
| 连接间隔 min | **20 ms**（0x10 × 1.25 ms） | 连接事件 50 次/s |
| 连接间隔 max | **40 ms**（0x20 × 1.25 ms） | 连接事件 25 次/s |
| GATT 特征最大长度 | **512 B** | `BLE_SLECG_TX_MAX_LEN` |

### 2.1 理论峰值吞吐

```
最佳情况：50 events/s × 514 B/event ≈ 25.7 KB/s
保守估计（40 ms 间隔）：25 events/s × 514 B ≈ 12.8 KB/s
```

实际可用带宽受协议开销、多包竞争、主机调度影响，建议按 **60~70%** 利用率规划，即约 **8~18 KB/s** 有效载荷。

## 3. ECG 组包方案

### 3.1 设计目标

- 满足 500 Hz 全量传输
- 控制 Notify 频率，降低协议开销与 CPU 中断负担
- 单帧远小于 MTU 与 512 B 上限，留余量给其他包类型

### 3.2 推荐参数

| 参数 | 值 | 推导 |
|------|-----|------|
| 每包样本数 `n_samples` | **25** | 500 Hz ÷ 25 = 50 ms 窗口 |
| ECG Notify 频率 | **20 包/s** | 500 ÷ 25 = 20 |
| 样本数据量/包 | 50 B | 25 × 2 B |
| PAYLOAD 总长 | **58 B** | seq(2)+ts(4)+n(1)+loff(1)+samples(50) |
| 完整帧长 | **66 B** | 信封 8 B + PAYLOAD 58 B |

> 注：计划中估算 ~71 B 含可选扩展；当前规范 PAYLOAD=58 B，帧长=66 B。

### 3.3 ECG 有效吞吐验算

```
样本数据：20 包/s × 50 B = 1000 B/s  ✓ 等于 500 Hz × 2 B
协议开销：20 包/s × 16 B（信封+头字段）= 320 B/s
ECG 合计：≈ 1320 B/s
```

## 4. 其他上行包占用

| 包类型 | 频率 | PAYLOAD | 帧长 | 带宽 |
|--------|------|---------|------|------|
| DEVICE_STATUS | 1 Hz | 12 B | 20 B | 20 B/s |
| BATTERY_ADC | 0.2 Hz | 5 B | 13 B | 2.6 B/s |
| IMU_DATA（预留） | 50 Hz | 20 B | 28 B | 1400 B/s |
| ACK/NACK | 按需 | 2 B | 10 B | 可忽略 |

### 4.1 总上行估算

| 场景 | 估算带宽 |
|------|----------|
| 仅 ECG 采集 | ≈ **1.3 KB/s** |
| ECG + STATUS + BATTERY | ≈ **1.35 KB/s** |
| ECG + STATUS + BATTERY + IMU（全启用） | ≈ **2.75 KB/s** |

对比 BLE 保守可用带宽 **8~18 KB/s**，余量充足（>60%）。

## 5. 组包参数备选方案

| 方案 | n_samples | 包频率 | 帧长 | 延迟 | 适用 |
|------|-----------|--------|------|------|------|
| **A（推荐）** | 25 | 20 Hz | 66 B | 50 ms | 平衡延迟与开销 |
| B | 10 | 50 Hz | 36 B | 20 ms | 低延迟，开销略高 |
| C | 50 | 10 Hz | 116 B | 100 ms | 最低开销，延迟较高 |

方案 A 在 50 ms 显示延迟与 20 次/s Notify 开销之间取得平衡，适合实时 ECG 波形。

## 6. 下行指令带宽

下行指令体积极小（START 10 B、STOP 9 B），频率低（用户操作触发），可忽略。

## 7. 约束检查清单

| 检查项 | 结果 |
|--------|------|
| ECG 帧 ≤ 512 B | ✓ 66 B |
| ECG 帧 ≤ MTU-3 | ✓ 66 ≪ 514 |
| 500 Hz 样本无丢失（带宽） | ✓ 1000 B/s ≪ 可用带宽 |
| 20 Notify/s ≤ 50 连接事件/s | ✓ |
| IMU 启用后总带宽 | ✓ 2.75 KB/s ≪ 8 KB/s |

## 8. 结论

在 500 Hz、int16 样本格式下，**每包 25 样本、20 包/s** 的 ECG 组包方案满足 BLE 传输需求，并为 STATUS、BATTERY、IMU 预留充足带宽。后续若提高采样率或增加通道数，可重新评估 `n_samples` 或改用更大 MTU（BLE 5.0）。
